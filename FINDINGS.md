# FINDINGS — proshop_mern

Structured audit of the current codebase.
The review covered hotspots, edge cases, outdated dependencies, hardcoded behavior, dead code signals, security and validation risks, and missing tests around critical flows.
The table below keeps only the highest-risk findings, ordered by exploitability, blast radius, and likelihood in the current runtime.
Once a finding is fixed, the relevant row should be updated with a commit reference.

| # | Risk | Where | What | How to fix | Status |
|---|------|-------|------|------------|--------|
| 1 | 🔴 critical | [`backend/controllers/orderController.js::addOrderItems`](backend/controllers/orderController.js) | The order API trusts `itemsPrice`, `taxPrice`, `shippingPrice`, and `totalPrice` from the browser and persists them as authoritative values. A forged request can underpay or create impossible totals because the server never recomputes pricing from trusted product data. | Recalculate monetary fields on the server from product IDs and quantities, ignore client-supplied totals, and reject requests whose computed totals do not reconcile. | ✅ fixed |
| 2 | 🔴 critical | [`backend/routes/uploadRoutes.js::router.post`](backend/routes/uploadRoutes.js) | The upload endpoint is public, writes directly into the web-served `uploads/` directory, and has no file-size limit. Any anonymous client can use it for storage abuse, disk exhaustion, or repeated public file placement. | Require authenticated admin access, add `multer` size limits and explicit upload error handling, and reject unsupported payloads before writing to disk. | 🔴 not yet |
| 3 | 🟡 medium | [`frontend/src/store.js::module initialization`](frontend/src/store.js) | Store hydration calls `JSON.parse` on `cartItems`, `userInfo`, and `shippingAddress` at module load with no recovery path. One malformed `localStorage` value is enough to crash the application before React renders the first screen. | Centralize storage hydration behind a safe parse helper with `try/catch`, fall back to defaults, and clear corrupted keys when parsing fails. | 🔴 not yet |
| 4 | 🟡 medium | [`backend/controllers/orderController.js::updateOrderToPaid`](backend/controllers/orderController.js) | The payment update path assumes `req.body.payer.email_address` always exists. A partial or changed PayPal payload turns a recoverable validation problem into an unhandled server error during payment confirmation. | Validate the inbound payment payload before mutating the order and persist only the payment fields that are actually present. | 🔴 not yet |
| 5 | 🟡 medium | [`frontend/src/screens/PlaceOrderScreen.js::PlaceOrderScreen`](frontend/src/screens/PlaceOrderScreen.js) | The `Place Order` button uses `disabled={cart.cartItems === 0}`, which compares an array object to a number and never disables the button. The UI therefore allows an empty-cart submit path and relies on the backend to reject it. | Change the UI guard to `cart.cartItems.length === 0` and keep a matching backend validation path for missing or empty order arrays. | 🔴 not yet |
| 6 | 🟡 medium | [`package.json::dependencies`](package.json)<br>[`frontend/package.json::dependencies`](frontend/package.json) | Several runtime dependencies remain multiple major versions behind current support baselines: `react@16.13.1`, `react-scripts@3.4.3`, `axios@0.20.0`, and `multer@1.4.2`. Mongoose has been upgraded from `5.10.6` to `8.22.1` in the current working tree, but the verified need for the Node 24 OpenSSL workaround is still a concrete symptom of frontend runtime drift. | Continue isolating upgrades into separate characterization-backed changesets, starting with packages that affect Node compatibility, security maintenance, and the payment or upload paths. | 🟡 partially addressed; commit pending |
| 7 | 🟡 medium | [`backend/controllers/orderController.js`](backend/controllers/orderController.js)<br>[`backend/routes/uploadRoutes.js`](backend/routes/uploadRoutes.js)<br>[`frontend/src/store.js`](frontend/src/store.js) | The highest-risk flows identified in this audit have no regression coverage in the repository. Order pricing, payment payload validation, upload restrictions, and store hydration currently depend on manual verification only, which raises the chance of reintroducing the same defects during later cleanup. | Add focused tests around order creation validation, payment payload handling, upload authorization and limits, and safe `localStorage` hydration before broader refactors. | 🔴 not yet |
