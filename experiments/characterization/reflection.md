# Characterization Reflection

This experiment establishes a behavioral baseline for `calculateOrderPrices` before any production refactor. The main architectural takeaway is that the function does more than calculate totals: it also encodes undocumented business rules and type assumptions that effectively form part of the runtime contract. In particular, the current implementation applies free shipping only when `itemsPrice > 100`, charges shipping for an empty order, and relies on JavaScript coercion so that stringified numeric inputs are silently accepted. None of these rules are documented at the API boundary, which makes them easy to change accidentally during cleanup.

The characterization tests also expose that the function fails in a non-uniform way for malformed input. A missing `qty` produces a partially computed result where `shippingPrice` remains `100` but `taxPrice` and `totalPrice` become `NaN`, while a `null` item fails earlier with a `TypeError`. From a design perspective, this is a signal that calculation, validation, and error semantics are currently coupled in an unstable way. That does not justify changing the behavior inside a refactor by default; it just means future changes should separate intentional business rules from incidental JavaScript behavior and add validation at a clearer boundary. The refactored copy in this experiment preserves the observed outputs exactly, which is the correct outcome for a characterization exercise: improve readability and structure without silently redefining the legacy contract.
